# 第四单元 OO 学习与反思

这学期的 OO 课程到这里基本告一段落了。回顾四个单元，课程不仅是在考察代码能力，更是在引导我们改变开发习惯。本文主要围绕第四单元的图书馆系统，结合前三个单元的作业，对自己在架构设计、测试方法上的思维演进进行一个整体复盘，同时也会分享一下使用大模型辅助建模的实际体验。

## 一、 第四单元正向建模与开发

第四单元的任务是实现并不断扩展一个图书馆管理系统。随着迭代推进，需求一直在增加：从最初单纯的借书、还书和预约（hw13），到加入精品书架、阅读室和评分机制（hw14），最后又引入了比较繁琐的信用分、借阅期限以及续借规则（hw15）。在这一阶段，我认为正向建模的方法帮我理清楚了思路。

### 1. 一阶类图：构建系统骨架
动手敲代码之前，我们需要先在 StarUML 里完成一阶类图。它的主要意义在于限制开发初期的随意性，强制把核心业务解耦。针对复杂的题面，我首先尝试把核心实体和场地对象分离出来：

- **核心实体**：提取了 `Reader`（维护读者的状态等）和 `Book`（维护单本图书的状态和流转历史）。
- **业务场所**：分离出了 `Bookshelf`（普通书架）、`BorrowAndReturnOffice`（借还处）和 `AppointmentOffice`（预约处）。

一阶类图不仅明确了系统大概需要哪些类，还初步划定了各个类的职责。正是因为有了这层约束，我在后续写代码时，才没有把所有控制逻辑全塞给 `MainClass` 或某一个庞大的 `Manager` 类。这让初期的业务逻辑比较清晰。

### 2. 二阶类图：对齐实现与设计的偏差
随着编码的深入，一阶类图中一些设计肯定会需要修改。比如在 hw14 中加入精品书架和阅览室后，原本统一的书架类就显得不够用了，我只能在代码里拆分出 `TreasuredBookshelf`、`ReadingRoom` 以及 `RatingOffice` 来分散业务；到了 hw15，`Reader` 的状态需要同时兼顾借书数量、信用分和逾期情况，不得不补充大量新字段。

这时候二阶类图的价值就体现出来了。它要求我们把迭代后的实际架构重新反映到模型中去。通过完善二阶类图，比如细化 `LibraryManager` 中 `returnBook`、`borrowBook` 的参数列表，我尽量确保了 UML 模型能真实反映代码，而不是变成一份单纯的设计文档。

## 二、 架构设计与模型追踪

在第四单元的设计中，我的原则是尽量遵循单一职责和高内聚低耦合的原则。我将系统分为了这些层次：

### 1. 核心架构划分
1. **实体对象**（`Book`, `Reader`）：主要负责管理自身的数据状态。`Book` 包含 ISBN 和具体的副本信息；`Reader` 则把借书上限、信用分计算、逾期判断等规则封装在内部，这样可以极大减轻管理类的负担。
2. **业务场所对象**（`Bookshelf`, `TreasuredBookshelf`, `ReadingRoom`, `AppointmentOffice`, `BorrowAndReturnOffice`, `RatingOffice`）：承担各个场所内的具体查询与书籍转移操作。例如 `TreasuredBookshelf` 单独处理 B 类和高分图书的管理；`RatingOffice` 接收用户的评分。
3. **统筹中枢**（`LibraryManager`）：作为系统的调度中心，它有前面的各种场地实例和用户表。它的职责是统筹 `borrowBook`、`returnBook`、`pickBook` 等请求，并在闭馆阶段调用 `arrange()` 方法搬运书籍。

### 2. 追踪关系分析
为了通过评测，我花了不少时间在代码和模型的对齐上：
- **类与属性的映射**：UML 模型里的每一个核心类都在代码中有完全对应的 Java 类。比如 UML 中 `LibraryManager` 和 `Bookshelf`、`ReadingRoom` 的组合或聚合关联，在 `LibraryManager.java` 里都对应着成员变量的声明。
- **状态流转与交互的映射**：在画状态机和顺序图时，我重点描述了一本书在各个场馆之间的流转路线。在实际代码中，这些状态变化是由 `LibraryManager` 的方法来驱动的。假设要归还一本应该放在精品书架的书，`LibraryManager` 会先调用借还处的接收逻辑，在闭馆时把它从借还处取出来，存进 `TreasuredBookshelf`。顺序图里的每一次跨对象交互，都能在代码的方法调用中找到一致的函数。

## 三、 大模型辅助正向建模

在这一单元里，我尝试了用大模型来辅助生成 UML 的相关数据。它在处理繁琐的形式化转换时效率很高，但如果不加限制，也很容易出现理解偏差或者过度设计的问题。

### 1. 实际使用中的优缺点
- **效率提升**：要把多个 Java 方法的定义手动录到 StarUML 里是非常枯燥且易错的。我把这部分代码发给大模型，它能在很短时间内把它们翻译成标准的 UML 格式，这就搞定了不少体力活。
- **理解偏差**：如果直接把作业需求扔给大模型让它设计，它往往会自作聪明地加入一些复杂的设计模式，这就把问题复杂化了。另外，在生成顺序图时，它也经常无视“跨类调用的方法必须是 public”这个重要的代码规范。

### 2. 引导大模型的策略
经历了这些问题后，我总结了几条给大模型写提示词的经验：
1. **结构化提供背景**：先让模型提取出领域里的基本名词，确定好讨论的前提，别一开始就让它输出完整设计。
2. **设定严格的边界**：在让它生成交互逻辑时，必须把课程的特殊要求作为前提给它定死。比如明确说明：“不要使用策略模式或命令模式，直接在 `LibraryManager` 里面写分发方法”，“不同类之间相互调用时，对应的方法必须是 public”。这样能有效规范它的发挥。
3. **通过极端场景验证**：我会构思一些复杂的场景去测试它的逻辑。比如问它：“如果一个读者成功预约了一本 B 类图书，但在保留期内该读者因为另一本书逾期未还导致信用分降到 80 分以下，那他还能不能正常取走这本 B 类图书？”通过这种极端情况，可以促使大模型暴露出它的设计漏洞。

## 四、 架构设计思维的演进

回顾整个学期的作业，我觉得自己在架构设计上的思路经历了明显的转变：

1. **第一单元（表达式解析）**
   通过学习递归下降，我搭建了 `Lexer` 和 `Parser`，并设计了表达式的树形结构（由 `Expr`, `Term`, `Factor` 派生出 `FuncFactor`, `ExpFactor`, `ChoiceFactor` 以及 `Poly`, `Mono`）。这是我第一次体会到利用多态和树结构来解耦问题的好处。

2. **第二单元（多线程电梯）**
   这个单元加入了多线程机制，我实现了 `InputThread`、`DispatchThread` 和 `ElevatorThread`，并且在 hw7 里处理了容易冲突的双轿厢电梯（`ShadowElevator` 和 `ElevatorShaft`）。这个过程让我意识到，好的设计不仅仅是多建几个类，关键在于如何把策略和执行分离开。通过独立出 `DispatchStrategy` 和 `MoveStrategy`，电梯本身变成了只负责执行动作的载体，复杂的判断交给了调度器。同时我也摸索出了怎么给 `BaseQueue` 这类共享资源合理加锁，以避免死锁和轮询。

3. **第三单元（JML 社交网络）**
   这一单元主要围绕社交网络展开，我的关注点转移到了契约式编程上。JML 强调每个方法都要有清晰的前置和后置条件。在遵循契约的基础上，我也学着在架构里添加了一些辅助类（如 `NetworkHelper` 和 `IndexedList`），通过相关图算法进行空间换时间的优化，以应对大量查询可能带来的 TLE 问题。

4. **第四单元（UML 图书馆）**
   在最后的正向建模阶段，课程要求我们在真正编写业务逻辑前，先用类图分析静态结构，用状态图理清生命周期，用顺序图规划动态交互。这让我逐渐摆脱了拿到需求就直接写代码的习惯，开始用一种更系统的工程思路去开发。

## 五、 测试思维的演进

随着系统复杂度的提高，我的测试方法也跟着进行了升级：

1. **U1 自动化对拍测试**
   初期主要是依靠 Python 脚本调用 SymPy 生成大量随机表达式，然后和自己程序的输出进行对比。这算是比较典型的黑盒测试，用来找一些语法解析上的疏漏。

2. **U2 场景化并发施压**
   在多线程环境下，普通的随机测试很难暴露出死锁或者线程竞争的问题。我开始学着人为构造一些高压场景，比如在极短的时间窗口内，向同一楼层发送大量请求，或者针对双轿厢在换乘层的会车情况进行专门的压力测试。

3. **U3 基于规格的等价类划分**
   有了 JML 之后，测试就有了直接的参考依据。我开始根据 `requires` 和 `exceptional_behavior` 的说明去做等价类划分，专门针对各种抛出异常的情况、容器容量上限以及边界值编写 JUnit 测试。这让测试变得更有系统性。

4. **U4 状态追踪与长序列覆盖**
   图书馆系统的很多 Bug 不会在单次操作中暴露，而是需要经过一系列长流程后才会出现。所以我的测试重心变成基于状态机的路径覆盖测试。通过对照状态图，我设计了各种包含跨日期信用分扣除、不同书架间书籍移动的长用例，重点检查对象的状态在多次运转后是否还能保持一致。

## 六、 课程收获总结

这学期的 OO 课程很有挑战性，带给我的收获不仅是 Java 语法的提升，更是开发思维的转变：

- **对复杂度的拆解能力**：从一开始写大而全的方法，到后来学会用抽象语法树解析文本、用分发器隔离并发逻辑、用 JML 明确函数边界、用 UML 规划系统结构，我逐渐掌握了该如何一步步把复杂需求拆解成可维护的代码。
- **更全面的工程意识**：我逐渐改掉了以前只为让程序跑通就行的习惯。现在在写代码前，我会下意识地去考虑“这个类的职责对不对”、“资源上锁的方式会不会出问题”以及“以后要是增加新功能，现在的代码改起来方不方便”。
- **排错与抗压能力**：无论是长时间排查电梯死锁问题、想办法优化图查询的性能，还是在画图时一遍遍修改 UML 细节，这些经历都锻炼了我的耐心和严谨度。

总而言之，经过这四个单元的练习，面向对象对我来说不再只是一个抽象的概念，而是变成了实际解决复杂工程问题的一套方法论。再见，2026 OO！感谢这段痛并快乐着的旅程，它将成为我程序生涯中无可替代的宝贵财富！
