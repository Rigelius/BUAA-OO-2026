## 第三部分：题目描述

在一所小型图书馆中，用户借阅图书需要遵守一定的规章制度。我们需要你模拟一个小型的图书管理系统，完成图书馆所支持的相关业务。

图书馆里的所有图书按照 **“类别号-序列号”** 的形式编制**ISBN号**（同学们可以理解为 ISBN 国际标准书号的简化形式，即一本书的ISBN号是唯一的，它的所有副本的ISBN号都是相同的）。图书分 A 、B 、C 三类，由类别号标识，每种类别可能包含多个图书。每个ISBN号的图书可能具有多个副本（具有不同副本号）。图书馆中具体的某一本图书由唯一的“**类别号-序列号-副本号**”形式编制**书号**。对于不同类别的书，有不同的借阅数量限制。

图书馆的运行分为两个时段：白天**开馆**，夜晚**闭馆**。开馆时，图书管理系统需要处理用户的**各种请求**，依据图书馆的**运行规则**决定是否批准用户的请求；闭馆时，图书馆内部依据需要**整理各部门的图书**，以满足接下来可能处理的各种请求。

在上次作业中，开馆时图书管理系统需要处理的请求包括：借书、还书、查询、预约、取书、阅读、归还和评分。

<font color = red>本次作业新增三项功能：1. 引入用户信用分系统（强化用户行为限制）；2. 增加图书借阅期限限制；3. 支持图书续订操作。</font>

### 一、流程描述

#### 重要概念说明

- **开馆**：
  - 是一个**时段**，指的是**从07:00到21:00**（24小时制）这一段时间。这个时段一般可以进行除**整理流程**外的各种流程。
- **闭馆**：
  - 是一个**时段**，指的是**从21:00到次日07:00**（24小时制）这一段时间。
- **开馆前整理**：
  - 是一个**时段**，指的是**从06:00到07:00**（24小时制）这一段时间。这个时段要进行一次**整理流程**。
- **闭馆后整理**：
  - 是一个**时段**，指的是**从21:00到22:00**（24小时制）这一段时间。这个时段要进行一次**整理流程**。
- **书架**
  - 是**普通书架**和**精品书架**的总称。
- **图书在架**
  - 是指**书架**持有对应ISBN号的任一副本。

#### 借阅流程

若一位用户来到图书馆借阅某图书，按照如下规则来进行处理：

1. 若欲借阅的图书无余本在架，借阅失败；
2. 若欲借阅的图书为 **A 类图书**，借阅失败（ A 类图书无法借阅）；
3. 若欲借阅的图书为 **B 类或 C 类图书**，若符合**图书借阅数量限制**，<font color = red>且**用户当前信用分大于 80（详见用户信用分系统部分）**</font>，则借阅成功，该用户从此刻起持有该书；否则借阅失败，图书仍留在**书架**。

##### 图书借阅数量限制

- A 类书：不可借阅，不可预约。
- B 类书：一人同一时刻仅能持有一个 B 类书的副本。
- C 类书：对于每一个ISBN号，一人同一时刻仅能持有一个具有该ISBN号的图书副本。

##### <font color = red>图书借阅期限限制</font>

图书的**借阅期限**规定如下：

- A类图书不可借阅，未定义其借阅期限。
- B类图书借阅期限为15天。
- C类图书借阅期限为30天。

若某书的借阅期限为 x 天，自用户拿到借阅书本（包括**借阅成功**和**取书成功**）的次日起算第 1 日，第 x 日当日还书不算逾期。例如借阅期限30天，用户于1月1日借阅成功，则1月31日还书不算逾期，2月1日还书算作逾期。

#### <font color = red>续订流程</font>

若一位用户需要续订其持有的某图书，按照如下规则进行处理：

1. 若该书已逾期，则**续订失败**；
2. 若未逾期，则该书借阅期限**延长7天**，即当前应还日期向后延长7天。

关于**借阅期限**的内容，详见**借阅流程**。

#### 还书流程

若一位用户来到图书馆归还某图书，则该用户会前往**借还处**，还书立即成功，从此刻起，借还处持有该书，用户不再持有该书。


#### 预约流程

若一位用户需要预约某图书，则该用户需要前往**预约处**登记。

1. 该用户必须满足下述要求才可以预约成功：
   - 预约的书不是 A 类书（规定 A 类书无法预约）。
   - 若已经持有一本 B 类书，则无法再预约任何 B 类书。
   - 若已经持有**某ISBN号**的 C 类书，则不能预约**该ISBN号**的 C 类书。
   - 若此前已经预定过图书且**还未成功取书**，则预约失败。
   - <font color = red>若用户**当前信用分不大于 80**，则预约 B/C 类图书失败（详见用户信用分系统部分）。</font>
2. 预约成功后，图书馆可在此后的整理流程中选择将书送至预约处，送至预约处的图书需要指定是为了满足哪个用户的预约请求而送达的，具体输出格式见后。
3. 图书送至预约处后，若在**开馆前整理**中送达，则从**当日**起为该用户保留 **5** 天；若在**闭馆后整理**中送达，从**次日**起为该用户保留 **5** 天。在此期间，该书不能在整理流程中移动。保留期第 5 天闭馆后，该书不再视作为该用户保留，视为**预约失效**。该书**自该次闭馆后整理起**，图书馆可对该书进行整理。从**次日**起，预约该书的用户将无法取走该书。
   - 例如，1月1日开馆前整理中送达，则1月5日闭馆后，该书不再为用户保留；1月1日闭馆后整理中送达，则1月6日闭馆后，该书不再为用户保留。
   - “**预约失效**”概念说明：指预约的图书送达预约处后，超过规定的保留期限，该书不再为用户保留的状态。此时，用户的预约状态被取消，图书馆可对该书进行整理，用户无法再取走该书。

#### 取书流程

用户只能去预约处取书。

1. 该用户必须满足下述要求才可以取书成功：
   - 预约处存在一本为该用户预留的图书。
   - 该书ISBN号与用户**要取图书的ISBN号相同**。
   - 取书后该用户持有的书仍然**满足借阅数量限制**。
2. 该用户取书成功后，由用户持有该书，且该用户对于该书的一个成功预约视作完成，不再生效。
3. 取书时，对于用户要取的这本书，如果预约处存在多本副本，只能取走**整理操作时指定给该用户预留**的图书副本。
4. **若取书失败**，图书不发生移动。

**<font color = red>值得注意的是，用户取书成功与借阅成功一样，都会产生借阅期限，仍然需要满足图书借阅期限限制，详见借阅流程。</font>**

#### 阅读流程

若一位用户来到图书馆**阅读**某图书，按照如下规则来进行处理：

1. 若欲阅读的图书无余本在架，阅读失败；

2. 若该用户存在当日阅读后未归还的图书，阅读失败；

3. 若欲阅读的图书在架，则该用户会从**书架**上取走图书，前往**阅览室**阅读，该书视为从普通书架/精品书架移动到阅览室，**不能再被任何人借阅**，用户阅读成功。

4. <font color = red>若阅读 B/C 类图书，用户**信用分必须大于 0**；若阅读 **A 类**图书，用户**信用分必须大于 40**，否则阅读失败。（详见用户信用分系统部分）</font>

#### 归还流程

若一位用户在当日已经成功阅读了某图书，在当日闭馆前可以选择归还，则该用户会前往**借还处**，阅读归还立即成功，从此刻起，**借还处**持有该书，**阅览室**不再持有该书。

**注意**：用户可以选择不主动归还图书，图书始终留在阅览室直到整理流程中被归还到书架。

#### 评分流程

若用户要对某ISBN号进行评分，可前往**评分处**进行评分，可选项**有且仅有0**(分)、**1**(分)、**2**(分)、**3**(分)、**4**(分)、**5**(分)六种。

- 某ISBN号的得分是所有对其评分的**平均值**（向下取整）。
- 所有ISBN号初始得分为0分。

#### 精品书架

ISBN号得分**不小于4分**的所有副本视为精品图书，其余图书为非精品图书。每次**开馆前整理**后，精品图书都应移动或保留至**精品书架**，非精品图书移动或保留至**普通书架**。精品图书仍可被借阅、预约。

+ 例如，1月2日这一天，A-0001首次被评分为5分，下一次有输入的开馆时间为1月5日，且从1月2日开始到1月5日之前该ISBN号仅被评分过这一次，则2026-01-05的开馆整理后，所有ISBN号为A-0001的副本都应被放到**精品书架**，其他非精品图书都应被放到**普通书架**。

+ 如果没有图书被视为精品图书，则本次开馆整理后精品书架为空。

#### 查询流程

若用户需要查询图书馆内某本书的历史移动轨迹信息，可前往**自助查询机**处查询：

- 输入“**类别号-序列号-副本号**”，查询某**副本**的历史移动轨迹。
- 可移动的状态包括：**普通书架**、精品书架、阅览室、借还处、预约处、用户。

<font color = red>额外地，用户也可以查询自身当前的信用分。</font>

#### 整理流程

每次**开馆前整理**和每次**闭馆后整理**，系统都有一次机会整理各处的图书。整理的目的是为了满足请求，此时系统可以在不违反**图书移动限制**的情况下移动图书馆内的可在整理流程中移动的图书。

我们对整理流程中图书的移动进行如下限制：

1. 在一次整理流程中，每本图书仅能**移动一次**（**终点为预约处**的移动不计入次数限制）；
2. 在**开馆前整理**后，借还处不应该有书，预约处不应该有逾期的书，阅览室不应有书；
3. 不可以为没有预定特定图书的用户预留该图书（即为该用户将该图书送至预约处）；
4. 自由考虑使用何种策略为预约预留图书（没有特别限制，但建议以提高借阅成功率为目的），但是不允许无条件拒绝为预约预留图书。

##### 图书移动限制

任何可以**引起图书位置变化**的操作，该书数量的变化**符合实际**（一次移动仅可以改变单一副本的位置，本次作业中**任何时候图书总数不变**）。

具体的，此次作业中，可以引起图书位置变化的操作如下：

* 用户借阅书架图书成功：普通/精品书架  -> 用户
* 用户取书：预约处 -> 用户
* 用户还书：用户 -> 借还处
* 用户阅读：普通/精品书架 -> 阅览室
* 用户归还：阅览室 -> 借还处
* 整理时的图书转运：借还处，普通书架，精品书架，预约处，阅览室这五个位置中从一个位置到另一个位置

#### <font color = red>用户信用分系统</font>

所有用户的初始信用分为100，上限为180，下限为0，即：

- 积分增加时，更新后信用积分=min(更新前信用积分+x，180)
- 积分减少时，更新后信用积分=max(更新前信用积分-x，0)

可以引起信用分变化的情况如下：

+ 加分

  1. 借阅期限内还书（包括借书和取书成功后的还书）立即+10分。
  2. 阅读后**当日主动归还**（即输入该用户的归还指令）立即+10分。
+ 减分

  1. 逾期还书：在该书**借阅期限的最后一日结束后**，若用户仍未还书，该用户信用积分**立即-15**。

  2. 阅读之后不还：当日闭馆后立即-10分。

  3. 预约后不取：用户每次预约成功，且图书送到预约处后，若用户在规定时间内未能取走该书，则当日闭馆后立即-15分。

信用积分分级，不同的分级有不同的阅读/借阅/预约权限：

* **积分 > 0**：允许阅读B/C类图书
* **积分 > 40**：允许阅读A类图书
* **积分 > 80**：允许借阅或预约B/C类图书

## 第四部分：输入输出

### 一、程序输入

程序输入由**两部分**组成：**图书馆初始书目信息**和**用户请求动态信息**。

输入第一行为一个正整数 n，代表图书馆中共有 n 种不同的图书，1≤n≤100。

接下来 n 行每行对应一种书的详细信息，格式为“**类别号-序列号 副本数**”，例如 "B-0001 10" 代表 B 类下序号为 0001 的书（即ISBN号为B-0001的书）馆藏 10 个副本。序列号为**四位数字**。输入保证不存在相同的 “类别号-序列号”，不同类别号下可以存在相同的序列号，每个ISBN号的书最多 10 个副本。**副本号从1开始依递增顺序编号**，例如 "B-0001 10" 代表十本书的编号为 B-0001-01, B-0001-02, ..., B-0001-10。

接下来若干行每行对应一条动态信息，格式为"**[YYYY-mm-dd] OPEN**"和"**[YYYY-mm-dd] CLOSE**"之间的若干条"**[YYYY-mm-dd] <学号> <操作> <类别号-序列号[-副本号]>**"或"**[YYYY-mm-dd] <学号> <操作>**"，代表在日期为 [YYYY-mm-dd] 的当天，用户 <学号> 对图书 <类别号-序列号[-副本号]> 进行 <操作> 或查询个人信用积分，除开馆指令和闭馆指令，设评测给定的指令条数为m，1≤m≤200。

**请注意，初始数据点是给评测机的原始输入，并不等于同学们的程序实际接收到的输入。但我们保证同学们的程序接收到的输入，输入指令中操作的图书编号，一定在图书馆中存在**。


在评测数据中，所有请求指令都位于当日的开馆指令和闭馆指令之间，对于没有任何指令的日期，评测数据可能省略开闭馆指令。形式化的限制如下：

* 数据的所有指令时间严格不减（不可能出现时间倒流的情况）。
* 对于任意开馆指令 A ，存在且仅存在一条相同日期的闭馆指令 B ，且 B 位于 A 之后；对于任意闭馆指令 P ，存在且仅存在一条相同日期的开馆指令 Q ，且 Q 位于 P 之前。
* 对于任意请求指令 Q ，保证存在相同日期的开馆指令位于 Q 之前，保证存在相同日期的闭馆指令位于 Q 之后。

**具体的 <操作> 种类**：

- queried：查询。
- borrowed：借书。
- ordered：预约书。
- returned：还书。
- picked：取书。
- read：阅读。
- restored：归还。
- graded：评分。
- <font color = red>renewed：续订。</font>

所有请求指令均已按时间先后顺序排序。也即同一日期的全部信息，可理解为按照输入出现的顺序依次从早到晚发生。 **在图书馆第1天开馆前**，图书馆的所有书本均位于**书架**上。

所有输入事件在 [2026-01-01] 至 [2026-12-31] 之间发生，保证日期合法。

**输入信息展示如下**：

| 场景                                                | 输入                                                  |
| --------------------------------------------------- | ----------------------------------------------------- |
| 图书馆开馆时输入                                    | "[YYYY-mm-dd] OPEN"                                   |
| 图书馆闭馆时输入                                    | "[YYYY-mm-dd] CLOSE"                                  |
| 用户查询图书历史移动轨迹信息时输入                  | "[YYYY-mm-dd] <学号> queried <类别号-序列号-副本号>"  |
| <font color = red>用户查询个人信用积分时输入</font> | "[YYYY-mm-dd] <学号> queried credit score"            |
| 用户借书时输入                                      | "[YYYY-mm-dd] <学号> borrowed <类别号-序列号>"        |
| 用户预约时输入                                      | "[YYYY-mm-dd] <学号> ordered <类别号-序列号>"         |
| 用户还书时输入                                      | "[YYYY-mm-dd] <学号> returned <类别号-序列号-副本号>" |
| 用户取书时输入                                      | "[YYYY-mm-dd] <学号> picked <类别号-序列号>"          |
| 用户阅读时输入                                      | "[YYYY-mm-dd] <学号> read <类别号-序列号>"            |
| 用户归还时输入                                      | "[YYYY-mm-dd] <学号> restored <类别号-序列号-副本号>" |
| 用户评分时输入                                      | "[YYYY-mm-dd] <学号> graded <类别号-序列号> <分数>"   |
| <font color = red>用户续订时输入</font>             | "[YYYY-mm-dd] <学号> renewed <类别号-序列号-副本号>"  |

**以下是类图所有的评测标准：**

**一、类图基本信息的检查：**

* R1：类图正确性基本检验
  * 类图中的所有元素，除 direction 为 return 的 UMLParameter ，UMLAssociation ，UMLAssociationEnd ，UMLGeneralization ，UMLInterfaceRealization 外，其余元素的 name 字段不能为空。
  * 不能含有重名的类。
  * 不能有循环继承。
  * 任何一个类或接口不能重复继承另外一个类或接口。
  * 接口的所有属性和方法均需要为 public 。
* R2：类图与程序一致性检验（类内部对应关系）（Class 正确性检验）
  * 类图与程序能互相找到名字相同的类。
  * 类图中每个类的属性：考察它们是否与程序中类的属性的**名字、可见性、类型**一致。
  * 类图中每个类的方法：考察它们是否与程序中类的方法的**名字、可见性、返回值类型**一致。
* R3：类图与程序一致性检验（类内部对应关系）（Interface 正确性检验）
  * 与 R2 中检测 class 的条目一致。
* R4：类图与程序一致性检验（类内部对应关系）（Enum 正确性检验）
  * 与 R2 中检测 class 的条目一致。
  * 额外的，类图中所定义的**枚举项**要求与代码中相应枚举类所定义**枚举项**名称相同，且是一一对应。
* R5：针对关系的类图与程序一致性检查
  * 类图中的**继承、实现、关联**关系应和代码实现中的关系保持一致，即同时存在或同时不存在。

关于 UML 类图和程序一致性的检查，有以下说明：

* 对于所提交的 UML 类图和所提交程序
  * 设类图中的设计单元集合为 MC ，程序中的实现单元集合为 CC
  * “单元”包括 class ，interface ，enum class
* 对于 MC 中任意一个单元 Ma
  * Ma 中属性个数或方法个数不少于 1
  * CC 中一定存在一个单元 Ca 与 Ma 相对应，且 Ma 中属性数目不少于 Ca 中属性数目的 60% ，同时 Ma 中方法数目不少于 Ca 中方法数目的 60%

* 对于 MC 中任意两个单元Ma和Mb的关系 Rab
  * CC 中必然有对应的两个单元 Ca 和 Cb ，且 Ca 和 Cb 同样具有相同的关系 Rab ，反之亦然

* 在程序中，如若一个设计单元 Ca 的成员属性引用了设计单元 Cb ，我们认为 Ca 、Cb 之间存在关联关系。

### 状态图

本次作业中，我们要求同学们自行设计图书的状态并绘制状态图。状态图通常在**类图和代码框架设计后**、**具体代码实现前**进行设计，本次作业**仅需**在**第二阶段提交入口**中提交**一次**状态图即可。状态图的评测规则如下：

- R1：状态图正确性基本检验

  - 状态图中只能有一个起始状态和0到1个终止状态

  - 起始状态只能有外出迁移，终止状态只能有进入迁移
- R2：Trigger与Guard检验

  - 除了起始状态的外出迁移，所有状态转移中Trigger和Guard至少要存在一个

  - 任意一个迁移的Guard所涉及的变量，都必须是程序中某个类的成员变量

  - 从一个状态转移到不同状态的Guard条件必须互斥（无论Trigger是否相同）
  - **状态图中迁移的Trigger必须和程序中有@Trigger注解的方法一一对应**（具体要求见后）
- R3：状态转移路径检验
  - 任意一个状态S，必然存在一条迁移路径（起点为起始状态，终点为S）
  - 任意一个状态S，必然存在一条路径迁移到终止状态（如果有）
  - 针对每个状态S，从起始状态到S的所有简单路径都必须有解
    - 路径有解：该路径上所有转移的Guard在一起（逻辑与）有解
    - 简单路径：路径中所有迁移循环走0次或1次

**关于程序中注解的添加方法说明**

对于程序中涉及到状态转移的方法，请按如下格式添加注解（需要使用官方包）：

```java
@Trigger(from = "S1", to = { "S2", "S3" })
@Trigger(from = "S2", to = "S3")
void triggerMethod() {
    // some code here...
}
```

该示例表示了triggerMethod是一个触发器， 可以将状态从S1转移到S2或S3（根据某些其他条件（guard））， 也可以将状态从S2转移到S3。

**关于状态图中Trigger和程序中有@Trigger注解方法一一对应的说明**

- 出现在注解中`from`和`to`中的状态名称，必须与状态图中某个状态的name相同，约定起始状态名为`InitState`，终止状态名为`FinalState`。
- 注解中涉及到的任意一条状态转移路径，在状态图中，都存在一条路径，使得该路径的起始状态与终止状态和注解中的路径一致，trigger函数名与注解标注的函数名一致。
- 图中任意一个迁移的Trigger，都对应程序中的一个添加@Trigger注解的方法，使得Trigger名与方法名一致。

**关于Trigger方法和Guard条件的填写说明**

1. 对于Trigger方法，只需要填写成形如`funcName()`形式。**不需要填写所在类名和方法名中的参数**，以免造成评测失败。（在starUML软件中，括号容易在画布中填写失败，可以在右下方`EDITORS`中的`Properties`中填写括号）

2. **请注意，添加Trigger时，请右键Transition，点击add Trigger，修改Trigger的name字段，而非Transition的name字段**

   ![image-20230507090704316](https://laiang8086-test.oss-cn-beijing.aliyuncs.com/images/image-20230507090704316.png)

3. 对于Guard条件表达式，按一般的C语言条件表达式形式填写即可，请填写在Transition的guard字段。其具体形式化约束符合如下的文法：

```bash
   条件表达式 Cond → LOrExp
   逻辑或表达式 LOrExp → LAndExp | LOrExp '||' LAndExp
   逻辑与表达式 LAndExp → EqExp | LAndExp '&&' EqExp
   相等性表达式 EqExp → RelExp | EqExp ('==' | '!=') RelExp
   关系表达式 RelExp → AddExp | RelExp ('<' | '>' | '<=' | '>=') AddExp
   加减表达式 AddExp → MulExp | AddExp ('+' | '−') MulExp 
   乘除模表达式 MulExp → UnaryExp | MulExp ('*' | '/' | '%') UnaryExp 
   单目运算符 UnaryOp → '+' | '−' | '!'
   一元表达式 UnaryExp → PrimaryExp | UnaryOp UnaryExp
   数值 Number → IntConst
   基本表达式 PrimaryExp → '(' Exp ')' | LVal | Number
   表达式 Exp → LOrExp
   左值 Lval 在本次作业中可以简单理解为变量名（当然在编译理论中它的含义并非如此）
   IntConst → DecimalConst | 0
   DecimalConst → NonzeroDigit | DecimalConst Digit
   NonzeroDigit → 1 | 2 | 3 | 4 | 5 | 6 | 7 | 8 | 9
   Digit → 0 | NonzeroDigit
```

1. 由于文法的限制，故对于对象的相等，简单用`==`代替`equals()`方法即可。
2. Guard条件中使用的变量忽略类型（无论是基本数据类型还是自定义类）。
3. 无论是Trigger方法还是Guard条件中涉及的变量，忽略其私有性问题。（即不区分`public`，`private`，`protected`）
4. Guard条件中不允许仅出现一个左值`Lval`。
5. 为避免Guard条件过于复杂导致可行域解算耗时太长，Guard条件表达式不得超过40个字符长度。
6. **同学们需要小心在画布上被删掉的状态仍然保留在模型框中。如果要删除的话一定要保证模型框（MODEL EXPLORER）中也删除完毕，不然可能出现未知的状态不可达。**

**温馨提示**

上述评测规则和填写说明，一方面是为了检查你的状态图**设计的合理性**，一方面是为了检查你的状态图和你的程序**这两者的一致性**。但同时应该明白，状态图本身仍然**以示意为主**，所以在设计状态图时，尽量不要在状态图中直接使用通常出现在程序代码中的语言字眼（比如在Guard条件中直接使用通常在代码中出现的if判断条件语句），这样可能会使你在完成状态图时遇到困难。事实上，你需要做的是具体分析源码中**哪些方法和变量体现了你写程序前的对状态和状态转移的设计**。如果不存在这样的方法和变量，你也可以在程序中合理定义一些较为过渡性的方法和中间变量来辅助表达。

### <font color = red>顺序图</font>

在本次作业中，我们要求同学们针对图书馆各个部门之间的交互行为及图书馆各个部门与用户之间的交互行为绘制顺序图。

在评测时，我们主要针对**预约和取书场景**进行正确性检验。我们规定，“用户成功预定新书”的起始消息应名为“**orderNewBook**”，“用户成功取书”的结束消息应名为“**getOrderedBook**”。

顺序图的评测规则如下：

- R1：对于上述预约和取书两个场景，顺序图中应存在相应的消息路径来实现该场景。
  - 消息路径：对于消息路径中任意相邻的两条消息，前一条消息的target必须为后一条消息的source。
- R2：Lifeline 的name应与代码中的某个类的名称相同。
  - Lifeline 命名规则："obj:class" | ":class"。class 对应代码中某个类名。
- R3：除 reply message 外，任何一个 message 的名称应与 source 所对应类或 target 所对应类的某个方法相同。
  - 如果消息的 source 和 target 所对应的类不同，消息名对应方法的可见性不能为 private。
- R4：任何一个 message 的 source 和 target 所对应的两个类如果不同，两个类之间应有关联关系。

##### Q：第一阶段作业入口未按时提交怎么办？

A：扣除一阶类图所有分数，在第二阶段提交时，不需要提交一阶类图的uml配置路径文件。

##### Q：第二阶段提交时，想对类图进行重构怎么办？

A：在第二阶段提交时，可以在完成二阶类图时进行重构，并提交重构说明文件，若助教审核通过（重构理由合理），则仍可以按评分细则正常得分。注意，助教审核是后期人工审核，不会立即反馈结果。

##### Q：如果第二阶段入口没有提交一阶类图的uml配置路径文件怎么处理？

A：第二阶段不进行相似度评测，第一阶段类图评分归零，其余正常评测。

##### Q：如果第二阶段入口没有提交二阶类图怎么处理？

A：不进行第二阶段类图所有评测，两阶类图评分均归零。
