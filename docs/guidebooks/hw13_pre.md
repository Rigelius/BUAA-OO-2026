## 第三部分：题目描述

在一所小型图书馆中，用户借阅图书需要遵守一定的规章制度。我们需要你模拟一个小型的图书管理系统，完成图书馆所支持的相关业务。

图书馆里的所有图书按照 **“类别号-序列号”** 的形式编制**ISBN号**（同学们可以理解为 ISBN 国际标准书号的简化形式，即一本书的ISBN号是唯一的，它的所有副本的ISBN号都是相同的）。图书分 A 、B 、C 三类，由类别号标识，每种类别可能包含多个图书。每个ISBN号的图书可能具有多个副本（具有不同副本号）。图书馆中具体的某一本图书由唯一的“**类别号-序列号-副本号**”形式编制**书号**。对于不同类别的书，有不同的借阅数量限制。

图书馆的运行分为两个时段：白天**开馆**，夜晚**闭馆**。开馆时，图书管理系统需要处理用户的**各种请求**，依据图书馆的**运行规则**决定是否批准用户的请求；闭馆时，图书馆内部依据需要**整理各部门的图书**，以满足接下来可能处理的各种请求。

在本次作业中，开馆时图书管理系统需要处理的请求包括：借书、还书、查询、预约和预约取书。

### 一、流程描述

#### 重要概念说明

- **开馆**：
  - 是一个**时段**，指的是**从07:00到21:00**（24小时制）这一段时间。这个时段一般可以进行除**整理流程**外的各种流程。
- **闭馆**：
  - 是一个**时段**，指的是**从21:00到次日07:00**（24小时制）这一段时间。
- **开馆前整理**：
  - 是一个**时段**，指的是**从06:00到07:00**（24小时制）这一段时间。这个时段要进行一次**整理流程**。
- **闭馆后整理**：
  - 是一个**时段**，指的是**从21:00到22:00**（24小时制）这一段时间。这个时段要进行一次**整理流程**。

#### 借阅流程

若一位用户来到图书馆借阅某图书，按照如下规则来进行处理：

1. 若欲借阅的图书无余本在架，借阅失败；
2. 若欲借阅的图书为 **A 类图书**，借阅失败（ A 类图书无法借阅）；
3. 若欲借阅的图书为 **B 类或 C 类图书**，若符合**图书借阅数量限制**则借阅成功，该用户从此刻起持有该书；否则借阅失败，图书仍留在**书架**。

##### 图书借阅数量限制

- A 类书：不可借阅，不可预约。
- B 类书：一人同一时刻仅能持有一个 B 类书的副本。
- C 类书：对于每一个ISBN号，一人同一时刻仅能持有一个具有该ISBN号的图书副本。

#### 还书流程

若一位用户来到图书馆归还某图书，则该用户会前往**借还处**，还书立即成功，从此刻起，借还处持有该书，用户不再持有该书。


#### 预约流程

若一位用户需要预约某图书，则该用户需要前往**预约处**登记。

1. 该用户必须满足下述要求才可以预约成功：
   - 预约的书不是 A 类书（规定 A 类书无法预约）。
   - 若已经持有一本 B 类书，则无法再预约任何 B 类书。
   - 若已经持有**某ISBN号**的 C 类书，则不能预约**该ISBN号**的 C 类书。
   - 若此前已经预定过图书且**还未成功取书**，则预约失败。
2. 预约成功后，图书馆可在此后的整理流程中选择将书送至预约处，送至预约处的图书需要指定是为了满足哪个用户的预约请求而送达的，具体输出格式见后。
3. 图书送至预约处后，若在**开馆前整理**中送达，则从**当日**起为该用户保留 **5** 天；若在**闭馆后整理**中送达，从**次日**起为该用户保留 **5** 天。在此期间，该书不能在整理流程中移动。保留期第 5 天闭馆后，该书不再视作为该用户保留，视为**预约失效**。该书**自该次闭馆后整理起**，图书馆可对该书进行整理。从**次日**起，预约该书的用户将无法取走该书。
   - 例如，1月1日开馆前整理中送达，则1月5日闭馆后，该书不再为用户保留；1月1日闭馆后整理中送达，则1月6日闭馆后，该书不再为用户保留。
   - “**预约失效**”概念说明：指预约的图书送达预约处后，超过规定的保留期限，该书不再为用户保留的状态。此时，用户的预约状态被取消，图书馆可对该书进行整理，用户无法再取走该书。

#### 取书流程

用户只能去预约处取书。

1. 该用户必须满足下述要求才可以取书成功：
   - 预约处存在一本为该用户预留的图书。
   - 该书ISBN号与用户**要取图书的ISBN号相同**。
   - 取书后该用户持有的书仍然**满足借阅数量限制**。
2. 该用户取书成功后，由用户持有该书，且该用户对于该书的一个成功预约视作完成，不再生效。**若取书失败**，图书不发生移动。
3. 取书时，对于用户要取的这本书，如果预约处存在多本副本，只能取走**整理操作时指定给该用户预留**的图书副本。

#### 查询流程

若用户需要查询图书馆内某本书的历史移动轨迹信息，可前往**自助查询机**处查询：

- 输入“**类别号-序列号-副本号**”，查询某**副本**的历史移动轨迹。
- 可移动的状态包括：书架、借还处、预约处、用户。

#### 整理流程

每次**开馆前整理**和每次**闭馆后整理**，系统都有一次机会整理各处的图书。整理的目的是为了满足请求，此时系统可以在不违反**图书移动限制**的情况下移动图书馆内的可在整理流程中移动的图书。

我们对整理流程中图书的移动进行如下限制：

1. 在一次整理流程中，每本图书仅能**移动一次**（**终点为预约处**的移动不计入次数限制）；
2. 在**开馆前整理**后，借还处不应该有书，预约处不应该有逾期的书；
3. 不可以为没有预定特定图书的用户预留该图书（即为该用户将该图书送至预约处）；
4. 自由考虑使用何种策略为预约预留图书（没有特别限制，但建议以提高借阅成功率为目的），但是不允许无条件拒绝为预约预留图书。

##### 图书移动限制

任何可以**引起图书位置变化**的操作，该书数量的变化**符合实际**（一次移动仅可以改变单一副本的位置，本次作业中**任何时候图书总数不变**）。

具体的，此次作业中，可以引起图书位置变化的操作如下：

* 用户借阅书架图书成功：书架  -> 用户
* 用户取书：预约处 -> 用户
* 用户还书：用户 -> 借还处
* 整理时的图书转运：借还处，书架，预约处这三个位置中从一个位置到另一个位置

## 第四部分：输入输出

### 一、程序输入

程序输入由**两部分**组成：**图书馆初始书目信息**和**用户请求动态信息**。

输入第一行为一个正整数 n，代表图书馆中共有 n 种不同的图书，1≤n≤100。

接下来 n 行每行对应一种书的详细信息，格式为“**类别号-序列号 副本数**”，例如 "B-0001 10" 代表 B 类下序号为 0001 的书（即ISBN号为B-0001的书）馆藏 10 个副本。序列号为**四位数字**。输入保证不存在相同的 “类别号-序列号”，不同类别号下可以存在相同的序列号，每个ISBN号的书最多 10 个副本。**副本号从1开始依递增顺序编号**，例如 "B-0001 10" 代表十本书的编号为 B-0001-01, B-0001-02, ..., B-0001-10。

接下来若干行每行对应一条动态信息，格式为"**[YYYY-mm-dd] OPEN**"和"**[YYYY-mm-dd] CLOSE**"之间的若干条"**[YYYY-mm-dd] <学号> <操作> <类别号-序列号[-副本号]>**"，代表在日期为 [YYYY-mm-dd] 的当天，用户 <学号> 对图书 <类别号-序列号[-副本号]> 进行 <操作>，除开馆指令和闭馆指令，设评测给定的指令条数为m，1≤m≤200。

**请注意，初始数据点是给评测机的原始输入，并不等于同学们的程序实际接收到的输入。但我们保证同学们的程序接收到的输入，输入指令中操作的图书编号，一定在图书馆中存在**。


在评测数据中，所有请求指令都位于当日的开馆指令和闭馆指令之间，对于没有任何指令的日期，评测数据可能省略开闭馆指令。形式化的限制如下：

* 数据的所有指令时间严格不减（不可能出现时间倒流的情况）。
* 对于任意开馆指令 A ，存在且仅存在一条相同日期的闭馆指令 B ，且 B 位于 A 之后；对于任意闭馆指令 P ，存在且仅存在一条相同日期的开馆指令 Q ，且 Q 位于 P 之前。
* 对于任意请求指令 Q ，保证存在相同日期的开馆指令位于 Q 之前，保证存在相同日期的闭馆指令位于 Q 之后。

**具体的 <操作> 种类**：

- queried：查询。
- borrowed：借书。
- ordered：预约书。
- returned：还书。
- picked：取书。

所有请求指令均已按时间先后顺序排序。也即同一日期的全部信息，可理解为按照输入出现的顺序依次从早到晚发生。 **在图书馆第1天开馆前**，图书馆的所有书本均位于**书架**上。

所有输入事件在 [2026-01-01] 至 [2026-12-31] 之间发生，保证日期合法。

**输入信息展示如下**：

| 场景                               | 输入                                                  |
| ---------------------------------- | ----------------------------------------------------- |
| 图书馆开馆时输入                   | "[YYYY-mm-dd] OPEN"                                   |
| 图书馆闭馆时输入                   | "[YYYY-mm-dd] CLOSE"                                  |
| 用户查询图书历史移动轨迹信息时输入 | "[YYYY-mm-dd] <学号> queried <类别号-序列号-副本号>"  |
| 用户借书时输入                     | "[YYYY-mm-dd] <学号> borrowed <类别号-序列号>"        |
| 用户预约时输入                     | "[YYYY-mm-dd] <学号> ordered <类别号-序列号>"         |
| 用户还书时输入                     | "[YYYY-mm-dd] <学号> returned <类别号-序列号-副本号>" |
| 用户取书时输入                     | "[YYYY-mm-dd] <学号> picked <类别号-序列号>"          |

+ 借书、预约等指令格式为<**类别号-序列号**>，代表**该ISBN号的书**
+ 查询、还书等指令输入格式为<**类别号-序列号-副本号**>，代表**该书号的某一本书**

**注意**：评测详情中的**stdin**不是完整的输入，使用官方包的同学可以从**stderr**获取你的程序接收到的完整输入，不使用官方包自行处理输入输出的同学可以根据需要向stderr打印信息，或根据输出反推输入。

当程序运行时发生表格中出现的场景时，要求输出相应信息。每条信息占一行。

**开馆时**：

**查询时**：

接收到查询指令后，需要输出一个整数k作为该图书的历史移动轨迹的条目数，接下来输出k行，每行一条图书的历史移动轨迹。

本次作业中，我们规定图书历史移动的所有可能经过的位置只能为如下四种：

- bookshelf (简写作 bs) ：书架
- borrow and return office (简写作 bro) ：借还处
- appointment office (简写作 ao) ：预约处
- user ：用户

**整理时**：

接收到开馆指令OPEN和闭馆指令CLOSE以后，需要输出一个整数k作为准备进行的图书移动次数，接下来输出k行，每行一条图书移动信息。（虽然**开馆前整理**的输出在"开馆指令OPEN"这条输入之后，但是我们仍认为在时间上，**开馆前整理**在开馆指令前。）

本次作业中，我们规定图书移动的起点和终点只能为如下三种：

- bookshelf (简写作 bs) ：书架
- borrow and return office (简写作 bro) ：借还处
- appointment office (简写作 ao) ：预约处

**注意：起点和终点不能相同**

| 场景                     | 输出                                                         |
| ------------------------ | ------------------------------------------------------------ |
| 图书移动，终点不为预约处 | "[YYYY-mm-dd] move <类别号-序列号-副本号> from <起点> to <终点>" |
| 图书移动，终点为预约处   | "[YYYY-mm-dd] move <类别号-序列号-副本号> from <起点> to <终点> for <学号>" |

图书转运路径的输出，应当紧接在开馆后和闭馆后。

终点为预约处的整理操作输出的副本号可以满足其他条件下任意指定，**后续对应的成功取书输出的副本号应与之相同**。

### 三、评测保证

本单元对程序正确性的检查采用**交互式评测**。具体的，我们的测试数据点只给出**借阅请求**，**预约请求**和**查询请求**，而**还书请求**与**取书请求**依据同学们的输出情况动态生成，以检测同学们的程序是否能够正确处理不同情况的请求。

* 评测机保证一定是针对拥有某书的相应用户来生成该书的还书请求。对每条成功的借阅请求和成功的取书请求，**至多生成 1 条**对应的还书请求。
* 评测机保证一定是某个用户对某书的预约成功之后才生成该书的取书请求，但不确定该请求发出时**相应的图书是否送达预约处**，以及**相应的预约是否失效**。对每条预约请求，**至多生成 2 条**取书请求。

## 第五部分：类图评测标准

### 类图

<font color = red>类图需要进行**两个阶段**的提交和评测：</font>

- **第一阶段**需要提交一阶类图和类名输出文件。**一阶类图**是在代码编写前进行架构设计时完成，目的是帮助同学们学会利用类图进行正向化建模。第一阶段评测内容为对一阶类图的**类图正确性基本检验(R1)**和**语义完整性检查**。一阶类图命名为“**uml_pre.mdj**"，提交到“**第十三次作业（第一阶段）**”的评测部分，**开放和截止时间均在代码开始提交之前**（具体请见评测部分）。类名输出文件内容如下，命名为**Main.java**。本次提交后会返回**一阶类图的uml配置路径**，将路径写到文件中，并命名为**config.yml**，用于第二阶段的提交（具体要求见下）。

  ```java
  //类名输出文件内容样例，其中输出内容需要替换为自己设计的所有类名，用英文逗号隔开
  public class Main {
      public static void main(String[] args) {
          System.out.println("MainClass,LibraryManager,User");
      }
  }
  ```

- **第二阶段**需要提交一阶类图的uml配置路径、二阶类图及其他文件（具体要求见指导书顶部）。**二阶类图**是在代码完成过程中对于一阶类图进行相应修正，目的是在代码具体实现过程中保持类图和代码的一致性。第二阶段评测内容为对二阶类图的**R1-R5的全面检验**和**语义完整性检查**，以及两阶类图的**相似度检查**。二阶类图命名为“**uml_ultimate.mdj**"，提交到“**第十三次作业（第二阶段）**”的评测部分，与代码、一阶类图的uml配置路径文件config.yml及其他文件一起提交。注意，若二阶类图有重构情况，需要**额外提交重构说明文件**。

**以下是类图所有的评测标准：**

**一、类图基本信息的检查：**

* R1：类图正确性基本检验
  * 类图中的所有元素，除 direction 为 return 的 UMLParameter ，UMLAssociation ，UMLAssociationEnd ，UMLGeneralization ，UMLInterfaceRealization 外，其余元素的 name 字段不能为空。
  * 不能含有重名的类。
  * 不能有循环继承。
  * 任何一个类或接口不能重复继承另外一个类或接口。
  * 接口的所有属性和方法均需要为 public 。
* R2：类图与程序一致性检验（类内部对应关系）（Class 正确性检验）
  * 类图与程序能互相找到名字相同的类。
  * 类图中每个类的属性：考察它们是否与程序中类的属性的**名字、可见性、类型**一致。
  * 类图中每个类的方法：考察它们是否与程序中类的方法的**名字、可见性、返回值类型**一致。
* R3：类图与程序一致性检验（类内部对应关系）（Interface 正确性检验）
  * 与 R2 中检测 class 的条目一致。
* R4：类图与程序一致性检验（类内部对应关系）（Enum 正确性检验）
  * 与 R2 中检测 class 的条目一致。
  * 额外的，类图中所定义的**枚举项**要求与代码中相应枚举类所定义**枚举项**名称相同，且是一一对应。
* R5：针对关系的类图与程序一致性检查
  * 类图中的**继承、实现、关联**关系应和代码实现中的关系保持一致，即同时存在或同时不存在。

关于 UML 类图和程序一致性的检查，有以下说明：

* 对于所提交的 UML 类图和所提交程序
  * 设类图中的设计单元集合为 MC ，程序中的实现单元集合为 CC
  * “单元”包括 class ，interface ，enum class
* 对于 MC 中任意一个单元 Ma
  * Ma 中属性个数或方法个数不少于 1
  * CC 中一定存在一个单元 Ca 与 Ma 相对应，且 Ma 中属性数目不少于 Ca 中属性数目的 60% ，同时 Ma 中方法数目不少于 Ca 中方法数目的 60%
* 对于 CC 中的任意一个单元 Ca
  * MC 中必然存在一个单元 Ma 与之相对应
* 对于 MC 中任意两个单元Ma和Mb的关系 Rab
  * CC 中必然有对应的两个单元 Ca 和 Cb ，且 Ca 和 Cb 同样具有相同的关系 Rab ，反之亦然

对于关联关系的补充说明：

* 在程序中，如若一个设计单元 Ca 的成员属性引用了设计单元 Cb ，我们认为 Ca 、Cb 之间存在关联关系。
* 在 UML 类图中的关联关系，既可以通过设计单元的属性引用隐式描述（属性类型是设计单元），也可以用 UMLAssociation 来显式描述。
* 建议在画图时，将重要的关联关系使用 UMLAssociation 进行描述，更便于理解设计单元之间的关系。

**二、语义完整性检查**

1. **关键词覆盖率**：为了保证所设计类图内容有效，我们提供以下词库，希望同学们**参考词库中的关键词**来设计类图的整体架构，**尽量覆盖关键词**；

   ```python
   key_words = {"borrow", "return",  "order", "pick", "query", "arrange", "library", "book", "bookcopy", "isbn", "open", "close", "bookshelf", "appointment office", "borrow and return office", "limit", "move", "user", "book category", "moving trace" }
   ```

2. **核心类设计**：对于题目描述中**涉及较多信息的主要实体**（如用户、图书、地点等），我们希望同学们设计对应的类，类图中**尽量包含所有核心类**；

3. **核心类的属性与方法设计**：对于题目描述中**主要实体对应的关键信息**，我们希望同学们在对应核心类中设计对应**属性或方法**。我们会检查上述词库中的关键词所在位置是否合理，即每个关键词需要出现在**适当的类名、类的属性名或方法名**中。该部分设计可参考下方提示的**重点考查关系**，并鼓励同学们在此基础上进行补充完善。

**提示：**

1. 可以考虑建立一个类专门处理各个请求，管理各个地点类；地点类要尽量包含对应的功能关键词。
2. **重点考查关系**：
   - 图书馆的地点类属性，图书馆的移动、管理、开闭馆方法；
   - 各个地点对应的处理请求方法；
   - 图书的ISBN号属性。

**三、两次类图相似度检查**

为了防止第一阶段提交一阶类图时存在提交“空图”等偷懒行为，我们规定**两阶类图差异不得过大**，并会进行相似度检查。若二阶类图存在重构情况，请与二阶类图一起**提交重构说明文件**，命名为“**重构说明.md**”。

若两阶类图相似度低于60%，且未提交重构说明文件或重构说明文件不合格，**一阶类图不给分**；其他情况按评分细则正常给分。

### **提示与注意**

**文件保存**

* 请将所绘制的类图保存在 **uml_pre.mdj / uml_ultimate.mdj** 文件中，并放在提交的**根目录**下（比如说与通常存放 Java 源代码 src 文件夹同级），否则将导致评测失败。
* 类名输出文件 **Main.java** ，一阶类图的uml配置路径 **config.yml** 和重构说明.md文件（若有），均放在提交的**根目录**下（比如说与通常存放 Java 源代码 src 文件夹同级），否则将导致评测失败。
* **请确保类图对应的 UMLModel 的 name 字段为 Model** ，否则会导致评测失败。

**使用 UML 进行设计**

* 完成作业时应先做好相应的设计，先绘制 UML 类图（**作为一阶类图提交在第一阶段入口处**），再实现具体代码。当然，代码变化后可以回头更新 UML 图（**作为二阶类图提交在第二阶段入口处**），但不应该在完成代码后直接照着代码来画 UML 图，那样就失去了设计建模的意义。
* 为了保证同学们写代码时的灵活性，在实现具体代码时，可以根据需要在代码中添加 UML 类图中未指明的**属性和方法**，但 UML 图中的属性和方法应在程序中进行实现，具体检查方法请见前文说明。**注意，不要为了画图方便刻意减少代码中类和方法的数量，也不要试图钻规则的空子，请同学们认真完成作业。**
* UML 建模工具一般都采用了模型和视图相分离的管理方式，即类图中看到的内容实际是对模型的观察。因此，在类图中简单删除一个要素并不一定导致模型发生变化。在完成本次作业过程中，建议同学们要注意阅读 mdj 代码检查各字段是否符合预期，例如检查是否存在仅删去视图未删去元素本身的情况。建议在右侧**模型框**（**MODEL EXPLORER**）中删除元素。
* 对于 UML 类图中的方法参数，请保证其在 mdj 文件中的先后顺序与程序中的一致。另外，除了构造方法，每一个方法的 RETURN 类参数应有且仅有一个。

##### Q：第一阶段作业入口未按时提交怎么办？

A：扣除一阶类图所有分数，在第二阶段提交时，不需要提交一阶类图的uml配置路径文件。

##### Q：第二阶段提交时，想对类图进行重构怎么办？

A：在第二阶段提交时，可以在完成二阶类图时进行重构，并提交重构说明文件，若助教审核通过（重构理由合理），则仍可以按评分细则正常得分。注意，助教审核是后期人工审核，不会立即反馈结果。

##### Q：如果第二阶段入口没有提交一阶类图的uml配置路径文件怎么处理？

A：第二阶段不进行相似度评测，第一阶段类图评分归零，其余正常评测。

##### Q：如果第二阶段入口没有提交二阶类图怎么处理？

A：不进行第二阶段类图所有评测，两阶类图评分均归零。
