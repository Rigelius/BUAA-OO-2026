## 第二部分：题目概况

本单元以某公司大楼的**目的选层电梯系统**为背景。

> 目的选层电梯系统的运行流程：
>
> 1. 进入电梯前，乘客输入目标楼层；
> 2. 系统根据设定的策略指定最合适的电梯来服务，将结果告知乘客（该功能将在hw6、hw7中要求，本次作业中不涉及）。
> 3. 乘客进入指定电梯后，无需再次按楼层按钮，电梯会自动将其送至目标楼层。

在三次迭代作业中，你需要根据需求与正确性约束，设计多线程程序来模拟该电梯系统的运行。

系统从标准输入中读入乘客请求信息（起点层、终点楼层），**请求调度器**会根据此时电梯运行状态将乘客请求合理**分配给某部电梯**（本次作业中不涉及分配，给每个乘客指定电梯）。被分配请求的电梯会经过**上下行，开关门，乘客进入/离开电梯**等动作将乘客从起点层运送到终点层。

请求的输入通过**课程组提供的**输入接口来**定时投放**请求，你需要调用课程组提供的接口进行输入输出。接口的具体细节见**第四部分-输入/输出接口**和**相关文档**。

**单部电梯的运行策略与电梯系统的调度策略均可任意指定**。也就是说，在任意时刻，电梯的上下行动、开关门操作都可以自定义，只要保证在**电梯系统运行时间不超过题目要求时间上限**的前提下将所有的乘客送至目的地即可。电梯运行的具体介绍见**第三部分**。

## 第三部分：电梯运行

### 功能概述

对于每个乘客请求，需要调度电梯将其完成，并将必要的运行信息通过输出接口进行输出。

具体而言，你需要控制电梯**上下行，开关门的动作**以及**控制乘客进出电梯**来将乘客从起点层运送到终点层。

在控制电梯运行的过程中需要**满足所有系统运行约束**，如运行速度、电梯载重等，运行约束的具体细节见本部分的**系统运行约束**一节。

在本次作业中，**每个乘客请求均指定了一部电梯负责接送该乘客**。你需要先输出一个`RECEIVE`信息表示电梯收到相应乘客请求，然后调度指定的电梯来接送对应乘客。

### 电梯系统默认参数

- 可到达楼层：地下B4-B1层，地上F1-F7层，共11层
- 初始位置：F1层
- 数量：6部
- 编号：6部电梯的`id`依次为1-6
- 移动速度：0.4s/层
- 额定载重量：400kg
- 开关门：开门到关门之间的间隔不小于0.4s

### 电梯载重说明

在现实情境中，电梯开门时允许轿厢内乘客暂时超过载重限制。当轿厢内乘客总体重超过载重时无法关门，此时会有乘客出电梯，等到符合载重限制才能出发。我们的作业要求与此相同，允许**开门时**暂时超过载重限制，只需保证**关门时**轿厢符合载重限制即可。

电梯系统的**全部6部电梯**载重均为400kg。

### RECEIVE约束说明

为避免出现多部电梯接送同一乘客造成资源浪费的情况，我们引入RECEIVE约束，要求在乘客被分配给某一电梯后输出相应`RECEIVE`信息。在本次作业中，每个乘客请求均指定了一部电梯负责接送该乘客。因此**在本次作业中**，对每个输入乘客请求**都按照其指定的电梯ID输出RECEIVE信息**。

电梯运行受到RECEIVE的约束，简单来说必须保证电梯在做出任何动作时都必须有未完成的RECEIVE。具体要求见本部分的**系统运行约束**一节。

### 系统运行约束：

电梯移动约束
- 电梯在两层楼之间移动应满足**移动时间约束**
- 电梯移动过程中**必须处于关门状态**
- 电梯只能在大楼的**楼层范围内**移动


电梯开关门状态约束
- **初始时**所有电梯为**关门状态**
- 程序运行**结束时**所有电梯必须处于**关门状态**
- 开门状态下才能关门，关门状态下才能开门
- 开门到关门之间的间隔不小于0.4s

RECEIVE约束
- 乘客进入电梯**必须**在输出该乘客的RECEIVE信息之后
  - 注意中途下电梯的乘客也需要重新输出RECEIVE信息才可上电梯
- 电梯移动**必须**在该电梯**当前有未结束的RECEIVE**时才能移动
  - 这里的移动指开始移动，即$t_{arrive时刻} - t_{每层移动时间}$时刻
  - 例如：电梯系统初始时所有电梯位于F1层，且没有未结束的RECEIVE。假如电梯系统在1.0s收到首个乘客请求，该乘客要从F2出发。则电梯最早在1.0s可以输出该乘客的RECEIVE信息，最早在1.0s可以开始向F2移动，最早在**1.4s**时可以输出首个ARRIVE信息。
- 程序**不能**RECEIVE**已经从目标层离开**的乘客，**不能**RECEIVE**上一次RECEIVE未结束**的乘客
- 程序输出某一乘客的RECEIVE信息时刻，相应的RECEIVE开始
- 程序输出某一乘客下电梯信息时刻，相应的RECEIVE结束
  - 在目标层或中途下电梯**均视为**RECEIVE结束

- **在本次作业中**，每个乘客的RECEIVE信息中的电梯id都**必须与指定的电梯id相同**。

乘客进出约束
- 乘客默认不在电梯里，电梯开门状态下才能进出电梯，且只能进入同层的电梯
- 乘客只有在电梯内才能出电梯，只有在电梯外才能进电梯
- 开门状态下乘客进出电梯**不花费**时间
- （本次作业）乘客**只能进入和离开**输入时指定的电梯，**不可以进入其他电梯**

乘客运送约束
- 系统运行中乘客**可以中途下电梯**，系统结束时到达目的地即可
- 系统结束时必须完成全部乘客请求
- 系统结束时**不能**有乘客困在电梯里

电梯载重约束
- 电梯为开门状态时，允许轿厢内乘客体重之**暂时大于**额定载重量
- 电梯为关门状态时，须满足轿厢内乘客体重之和**小于等于**额定载重量

运行时间约束
- 程序运行时间不能超过$T_{max}$。公测、强测、互测的$T_{max}$规则介绍见**第四部分-数据限制**。

输出限制
- 对每个乘客请求，均须输出相应RECEIVE信息
- 电梯每次移动后到达新楼层时须输出相应信息
- 电梯每次开门、关门时均须输出相应信息
- 乘客每次进入、离开电梯时均须输出相应信息
- 运行时的**任意时刻**须满足**输出时间戳非减限制**

7. **不需要对输入输出接口进行修改**。

### 输入格式

- 每个乘客的请求格式如下：`[时间戳]乘客ID-WEI-体重-FROM-起点层-TO-终点层-BY-指定的电梯ID`

### 输出格式

通过调用输出接口的每一次输出将自动附加时间戳于头部，具体请参考**输出接口文档**。

请**不要试图伪造时间戳**，否则会在实际评测时产生不一样的结果导致程序运行错误，由此产生的结果自行承担。

- 电梯接收分配：`[时间戳]RECEIVE-乘客ID-电梯ID`

- 电梯到达某一位置：`[时间戳]ARRIVE-所在层-电梯ID`

- 电梯开始开门：`[时间戳]OPEN-所在层-电梯ID`

- 电梯完成关门：`[时间戳]CLOSE-所在层-电梯ID`

- 乘客进入电梯：`[时间戳]IN-乘客ID-所在层-电梯ID`

- 乘客离开电梯：
  - 当前楼层**是乘客目标楼层**时：`[时间戳]OUT-S-乘客ID-所在层-电梯ID`
  - 当前楼层**不是乘客目标楼层**时：`[时间戳]OUT-F-乘客ID-所在层-电梯ID`

### 数据限制

#### 基本限制

- 输入电梯系统的指令数：**1~100条**（指令数为0或超过100均为不合法输入）。
- 每个乘客请求具有唯一不重复的`id`，`id`为`int`范围内的一个正整数。
- 每个乘客请求的体重为$[50,100]$范围内的一个正整数。
- 保证不出现起点层和终点层相同的乘客请求。
- 输入数据的时间戳精确到**小数点后一位**。

#### 公测数据限制

- 标程运行时间$T_{std}$：课程组标程运行测试点所使用的时间。标程使用ALS调度策略，具体细节见**第七部分-实现提示**。

- 公测中$T_{max}$与标程运行时间$T_{std}$的计算公式如下。
  $$
  T_{max} = \max\left(T_{std}+30, 1.3\times T_{std}\right)
  $$

- 包括强测在内的数据都会保证正常程序的电梯系统时间不会超过 $T_{max}$，也会**避免使用对边界情况较为敏感的数据**。
- **对于中测和强测数据，$T_{max}$ 将会严格限制为通过上述公式计算得到的值**。所以，请不要试图用超长的 sleep 来回避线程安全停止的问题。

#### 互测数据限制

- 系统时间上限 $T_{max}$：对于互测数据，$T_{max}$ 为120s。
- 第一条指令的投喂时间在1s或1s以后。
- 最后一条指令的输入时间不晚于50s。
- 指令条数不超过70。
- **同一部电梯最多有30条相关的乘客请求**。
- 随测试点提交的输出合法。

## 第六部分：性能分

**本单元每次作业总分100分，分为功能分85分，性能分15分。**

在强测时，性能分将由三部分组成：系统运行时间，平均完成时间和系统耗电量。

### 系统运行时间

系统运行时间$T_{run}=\max(T_{real},T_{final})$。其中，$T_{real}$为程序的总体运行时间（real time），$T_{final}$为最后一条输出对应的时间戳。

### 平均完成时间

平均完成时间$T_{avg}$的计算方法为：设共有$N$条乘客请求，第i位乘客的发出请求的时刻为$T_{i0}$，到达目标楼层的时刻为$T_{i}$，定义整个系统的平均完成时间为
$$
T_{avg} = \frac{\sum_i (T_i - T_{i0})}{N}
$$

### 系统耗电量

系统耗电量$W$将根据输出信息中的`ARRIVE`, `OPEN`, `CLOSE`信息的数量进行计算。

- 每条`ARRIVE`信息表示电梯移动一层，耗电量$W_{ARRIVE}=0.4$
- 每条`OPEN(CLOSE)`信息表示电梯开门（关门）一次，耗电量$W_{OPEN}=W_{CLOSE}=0.1$
- 系统的耗电量$W=W_{OPEN}N_{OPEN}+W_{CLOSE}N_{CLOSE}+W_{ARRIVE}N_{ARRIVE}$，其中$N_{OPEN},N_{CLOSE},N_{ARRIVE}$分别表示输出中$OPEN,CLOSE,ARRIVE$信息的总数。

### 性能分构成

设对于所有同学的数据，某一性能指标$x$的平均值为$x_{arg}$，最大值为$x_{max}$，最小值为$x_{min}$，则定义以下折算函数

$$
base_{min}= p \cdot x_{arg} + (1 - p) \cdot x_{min} \\\\
base_{max}= p \cdot x_{arg} + (1 - p) \cdot x_{max} \\\\
p = 0.10
$$

$$
r(x)= \begin{cases}1 & x \leq base_{min} \\\ \frac{base_{max} - x}{base_{max} - base_{min}} & base_{min} < x \leq base_{max} \\\ 0 & x > base_{max}\end{cases}
$$

最终性能分公式为
$$
s=15\times(0.3r(T_{run})+0.3r(T_{avg})+0.4r(W))
$$

性能分公式可能比较复杂，但实际上公式表达出的意思就是希望你的程序**在保证正确性的前提下**能够尽量做到以下三点：

- 运行时间尽量短
- 尽快完成每个乘客的请求
- 尽量减少系统的无效运行

## 第七部分：实现提示

1. 电梯的"运行策略"实际上可以抽象为从一组待选任务（如开关门、上下移动）当中选取一个来执行。可以先编写一个**等候乘客表类**来显示每个电梯各层等待电梯服务的乘客。再编写一个**电梯策略类**，接受这个候乘表类，分析并返回一个目标动作（开关门、上下移动等）。这样一来**电梯类**只需执行策略类返回的动作，将策略分析与动作执行解耦。

2. 电梯的运行流程可以抽象为：到达一层→开门→乘客进出→关门→寻找下一目标层→移动，可以使用状态模式建模。

   ![电梯运行逻辑](https://api-oo.e1.buaa.edu.cn/image/2024_03_22_12_36_14_abd2e2e4c2d1c4031cceb2800020e824587cfde2)

3. 在电梯的分派上，**本次作业**指定了完成乘客请求的电梯。**后续作业**同学们可以自行设计分派逻辑将乘客请求分配给合适的电梯。

  - 为了架构清晰、逻辑解耦以方便后续扩展，本次作业中可以先写一个**空白逻辑分派器**线程。输入线程将乘客请求发送给分派器线程，分派器再按照乘客请求中指定的电梯id将请求发送给相应电梯线程。
  - 尽管本次作业中分派器中并没有任何分派逻辑，只是按照输入要求进行分派，看似冗余。但从架构设计层面比较有益，在后续作业中分派器线程将在**乘客分派**与**线程同步控制**中发挥重要的作用。

4. 在**不违反课程规则**的前提下，我们对电梯的**捎带策略不做限制**。鼓励同学们积极查询资料并自行探索相关算法，对电梯的捎带策略有自己的设计与思考。

  - 为保证同学们实现的都为作业要求的**可捎带电梯**，我们对电梯的性能做一定的约束，采用**ALS调度策略**为性能**基准**计算$T_{std}$。对于系统中的每一个电梯，都采用 ALS 策略，即新增**主请求**和**被捎带请求**两个概念：
    - 主请求选择规则：
      1. 如果电梯中没有乘客，则从该电梯请求队列中选择第一个请求作为主请求。
      2. 如果电梯中有乘客，则从电梯内乘客容器中选择第一个请求作为主请求。
    - 被捎带请求选择规则：
      1. 电梯的主请求存在
      2. 该请求投喂的时刻**小于等于**电梯到达该请求出发楼层关门的截止时间
      3. 电梯的运行方向和该请求的目标方向一致
    
  - 请同学们合理设计自己的捎带策略，保证自己的电梯**性能符合基本要求**，即程序运行时间$T\leq T_{max}$，$T_{max}$的定义见**第四部分-数据限制-公测数据限制**。

  

## 第八部分：提示与警示

- **代码实现中不存在轮询**。轮询是一种非常占用 CPU 资源的行为，为了避免出现轮询的情况，**请使用 Wait-Notify 的方式编程**。同时，为了检查是否出现轮询的情况，总 CPU 时间限制为 10s，不满足该限制的程序会被判定为错误。

- 查看官方包接口的使用方法，可以在 IDEA 里面，按 Ctrl+Q ，可以看到类和方法的使用格式以及注释（本次官方提供的接口均提供了中文版 javadoc 注释，Ctrl+Q 可以直接查看）。

- 请**在代码一开始就初始化时间戳**，否则可能会出现和评测不一致的情况。

  ```java
  import com.oocourse.elevator1.TimableOutput;
  
  public class MainClass {
    public static void main(String[] args) {
      TimableOutput.initStartTimestamp();  // 初始化时间戳
  		/*
  			代码部分
  		*/
    }
  }
  ```

- 官方提供的输出包是**线程安全**的，同学们无需对其进行同步控制。

- 由于**多线程调度存在随机性**，代码中潜在的线程不安全问题不一定能稳定触发。因此，在使用官方提供的投喂脚本或进行评测环节时，可能会出现**输出与官方评测机不一致、bug 无法复现的情况**。最终结果均**以官方评测机为准**。

- **基于行断点机制的调试会对程序的运行行为产生较大影响，应谨慎使用**。
